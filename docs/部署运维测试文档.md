# 进销存管理系统 — 部署运维测试文档

## 一、系统概述

### 1.1 项目简介

本系统是一套基于 Spring Cloud 微服务架构的进销存管理系统（Inventory Management System），涵盖采购管理、销售管理、库存管理、订单管理、基础信息管理、财务管理、系统管理等核心业务模块。

### 1.2 技术架构

| 层次 | 技术选型 |
|------|----------|
| 前端框架 | Vue 3 + Vite 7 + Element Plus + Pinia |
| 后端框架 | Spring Boot 3.2.1 + Spring Cloud 2023.0.0 |
| 微服务治理 | Spring Cloud Alibaba 2023.0.1.0 |
| 服务注册与发现 | Nacos 2.x |
| API 网关 | Spring Cloud Gateway |
| 服务间调用 | OpenFeign + LoadBalancer |
| 分布式事务 | Seata 1.7 |
| 熔断降级 | Sentinel |
| 数据库 | MySQL 8.0 |
| ORM 框架 | MyBatis Plus 3.5.5 |
| 安全认证 | Spring Security + JWT |
| 容器化部署 | Docker + Docker Compose |
| 开发语言 | Java 17 / TypeScript |

### 1.3 微服务划分

| 服务名称 | 容器内部端口 | 宿主机默认端口 | 数据库 | 职责描述 |
|----------|--------------|----------------|--------|----------|
| user-service | 8081 | 不开放 | user_service_db | 用户注册/登录、JWT 认证、登录日志 |
| role-service | 8082 | 不开放 | role_service_db | 角色管理、权限管理（树形结构） |
| order-service | 8083 | 不开放 | order_service_db | 采购/销售订单、付款/收款记录 |
| inventory-service | 8084 | 不开放 | inventory_service_db | 产品/分类/仓库管理、出入库、库存台账 |
| base-service | 8085 | 不开放 | base_service_db | 客户、供应商、财务记录管理 |
| gateway-service | 9090 | 不开放 | — | API 网关，统一路由、鉴权、限流 |
| frontend | 80 | 127.0.0.1:18080 | — | Vue 3 前端应用 |
| nacos | 8848 | 127.0.0.1:18848 | — | 服务注册与发现控制台 |

### 1.5 微服务核心特性

- **服务注册与发现**：所有微服务启动后自动注册到 Nacos，通过服务名进行互相发现
- **统一网关入口**：Gateway 统一入口鉴权，按路径前缀路由到各服务（lb:// 负载均衡）
- **服务间远程调用**：order-service 通过 OpenFeign 调用 inventory-service（入库/出库联动）和 base-service（财务记录联动），含 FallbackFactory 降级
- **分布式事务**：跨服务操作（如采购入库涉及订单和库存两个数据库）通过 Seata 保障事务一致性
- **熔断降级**：Sentinel 控制流量和熔断，网关层返回统一降级响应 `{"code":429,"message":"服务繁忙，请稍后重试"}`
- **配置中心**：Nacos Config 支持配置热更新，无需重启服务

## 二、已实现功能清单

### 2.1 用户与权限

- 用户注册、登录（JWT Token 认证）
- 用户管理（增删改查、批量删除、状态管理）
- 登录日志查询
- 角色管理（增删改查、分配权限）
- 权限管理（树形结构、增删改查）
- 基于角色的访问控制（ADMIN/WAREHOUSE/SALES/PROCUREMENT/FINANCE）

### 2.2 采购与销售

- 新增采购订单（选择供应商、仓库、商品、**可选属性配置**）
- 新增销售订单（选择客户、商品、**可选属性配置**）
- 订单列表、详情、编辑、删除、批量删除
- 付款记录管理（付款后**自动更新订单 paymentStatus 和 orderDebt**）
- 收款记录管理（收款后**自动更新订单状态为"已收款"**）
- 采购/销售发票页面

### 2.3 库存管理

- 产品信息管理（属性支持多配置，逗号分隔，下单时下拉可选）
- 分类管理、仓库管理
- 入库操作（入库后**自动同步库存台账**）
- 出库操作（出库后**自动同步库存台账**，库存不足时拒绝）
- 库存记录查询
- 库存台账（按产品+仓库自动汇总，实时更新 finalStock）

### 2.4 基础信息与财务

- 客户管理、供应商管理
- 财务记录管理
- 统计报表看板

## 三、部署方案

### 3.1 环境要求

| 软件 | 版本要求 | 用途 |
|------|----------|------|
| JDK | 17+ | Java 运行环境 |
| Node.js | 18+ | 前端构建 |
| MySQL | 8.0+ | 数据库 |
| Docker | 24.0+ | 容器化部署 |
| Docker Compose | 2.20+ | 容器编排 |
| Maven | 3.8+ | 后端构建 |

### 3.2 数据库初始化

```bash
mysql -u root -p
source docs/user_service_db.sql;
source docs/role_service_db.sql;
source docs/order_service_db.sql;
source docs/inventory_service_db.sql;
source docs/base_service_db.sql;
```

### 3.3 本地开发启动

```bash
# 1. 启动 Nacos（单机模式）
docker run -d --name nacos -e MODE=standalone -p 8848:8848 -p 9848:9848 nacos/nacos-server:v2.2.3

# 2. 启动 Seata
docker run -d --name seata -e SEATA_PORT=8091 -e STORE_MODE=file -p 8091:8091 seataio/seata-server:1.7.1

# 3. 构建并启动后端服务
mvn clean package -DskipTests
cd backend/user-service && mvn spring-boot:run &
cd backend/role-service && mvn spring-boot:run &
cd backend/order-service && mvn spring-boot:run &
cd backend/inventory-service && mvn spring-boot:run &
cd backend/base-service && mvn spring-boot:run &
cd backend/gateway-service && mvn spring-boot:run &

# 4. 启动前端
cd frontend && npm install && npm run dev
```

### 3.4 Docker Compose 一键部署

```bash
# 构建所有后端服务 jar
mvn clean package -DskipTests

# 启动所有容器
docker-compose up -d

# 查看状态
docker-compose ps

# 查看日志
docker-compose logs -f gateway

# 停止
docker-compose down
```

默认只有前端和 Nacos 控制台开放到宿主机，避免占用本机数据库、后端服务等常用端口：

- 前端：`http://localhost:18080/`
- Nacos 控制台：`http://localhost:18848/nacos`

如果本机端口被占用，可临时改端口启动：

```bash
FRONTEND_PORT=28080 NACOS_PORT=28848 docker-compose up -d
```

### 3.5 服务启动顺序

```
MySQL → Nacos → Seata → 五个业务微服务（并行启动） → Gateway → Frontend
```

### 3.6 Gateway 路由规则

| 路由路径 | 目标服务 |
|----------|----------|
| `/api/auth/**` | user-service（白名单，无需鉴权） |
| `/api/users/**`, `/api/login-logs/**` | user-service |
| `/api/roles/**`, `/api/permissions/**` | role-service |
| `/api/orders/**`, `/api/reports/orders/**` | order-service |
| `/api/products/**`, `/api/stock/**`, `/api/inventory-ledger/**`, `/api/extra/**`, `/api/reports/stock/**` | inventory-service |
| `/api/base/**`, `/api/reports/finance/**` | base-service |

## 四、运维方案

### 4.1 日志管理

```bash
# 查看指定服务日志
docker logs -f ims-user-service --tail 200

# 导出日志
docker logs ims-order-service > order-service.log 2>&1
```

### 4.2 服务监控

- Nacos 控制台：`http://localhost:18848/nacos`（服务列表、配置管理）
- Sentinel 控制台：默认未在 Compose 中启动；需要时可单独部署
- Seata 默认不对宿主机开放；本项目使用 `seata:8091` 在 Docker 内网通信

### 4.3 容器管理

```bash
# 重启单个服务
docker-compose restart order-service

# 扩展实例（负载均衡测试）
docker-compose up -d --scale inventory-service=2

# 查看资源占用
docker stats

# 进入容器调试
docker exec -it ims-user-service /bin/sh
```

### 4.4 数据库备份

```bash
docker exec ims-mysql mysqldump -u root -p123456 --all-databases > backup_all.sql
```

### 4.5 配置中心热更新

1. 登录 Nacos 控制台 → 配置管理
2. 修改对应服务的配置（如 `warehouse.name`）
3. 发布配置后，服务自动加载，无需重启

## 七、测试结论

通过以上全面的测试验证：

1. **微服务架构的独立性**：各服务可独立部署、独立运行、独立扩缩容
2. **服务间通信的可靠性**：通过 OpenFeign 实现的跨服务调用稳定可靠，含 Fallback 降级
3. **分布式事务的一致性**：Seata 保障了跨服务数据操作的事务一致性
4. **系统的可扩展性**：通过 Nacos + 负载均衡，可动态扩展服务实例
5. **系统的容错性**：Sentinel 熔断降级机制保障了局部故障不扩散
6. **配置的灵活性**：Nacos 配置中心支持热更新，无需重启服务
7. **部署的便捷性**：Docker + Docker Compose 实现一键容器化部署
8. **业务数据联动**：收款后订单状态自动更新、入出库后库存台账自动同步
